# DeepShield AI v3

A production-ready web application that detects whether an uploaded image is
**REAL** or **FAKE** using a fine-tuned PyTorch **EfficientNet-B0** model
(`best_model-v3.pt`). No demo mode, no simulated predictions — every scan runs
real inference through your trained model.

## 2. Adding your model file

`best_model-v3.pt` is **not generated by this build** — you must place your
actual trained model file at:
backend/best_model-v3.pt

The backend (`app.py`) automatically handles three common save formats:
1. A full pickled `nn.Module` (`torch.save(model)`)
2. A raw `state_dict` (`torch.save(model.state_dict())`)
3. A checkpoint dict (`torch.save({"model_state_dict": ..., ...})`)

It rebuilds `torchvision.models.efficientnet_b0` with a 2-class classifier
head (`REAL` / `FAKE`) if a `state_dict` is detected.

**Class order:** `app.py` currently assumes:
```python
CLASS_NAMES = ["FAKE", "REAL"]   # index 0 = FAKE, index 1 = REAL
```
This matches the default alphabetical class order PyTorch's `ImageFolder`
assigns when your training data was organized as `fake/` and `real/`
subfolders. **If your model's classes were indexed differently, edit this
one line in `app.py` before deploying.**

## 3. Backend Deployment (Render)

### Step 1 — Push to GitHub
Push the entire `DeepShield-AI/` repo (including `backend/best_model-v3.pt`)
to a GitHub repository.

> ⚠️ If your model file is large (>100MB), use [Git LFS](https://git-lfs.github.com/)
> to track it, since GitHub blocks files over 100MB via normal git.
> ```bash
> git lfs install
> git lfs track "*.pt"
> git add .gitattributes
> git add backend/best_model-v3.pt
> git commit -m "Add model via LFS"
> ```

### Step 2 — Create a new Web Service on Render
1. Go to [render.com](https://render.com) → **New +** → **Web Service**
2. Connect your GitHub repo
3. Configure:
   - **Root Directory:** `backend`
   - **Environment:** `Python 3`
   - **Build Command:**
 pip install -r requirements.txt
   - **Start Command:**
 gunicorn -c gunicorn.conf.py app:app
   - **Instance Type:** at least `Standard` (512MB+ RAM) — PyTorch +
     EfficientNet-B0 needs more memory than Render's free tier `Starter` plan
     may reliably offer. Free tier can work for light testing but may be slow
     or run out of memory on cold starts.

### Step 3 — Deploy
Click **Create Web Service**. Render will build and deploy automatically.
Once live, your backend URL will look like:
https://deepshield-ai-v3.onrender.com

### Step 4 — Verify the backend
```bash
curl https://deepshield-ai-v3.onrender.com/
# Expected: {"status":"DeepShield AI v3 running"}

curl https://deepshield-ai-v3.onrender.com/health
# Expected: {"status":"ok","model_loaded":true, ...}
```
If `model_loaded` is `false`, check the Render logs — `model_error` in the
`/health` response (and the deploy logs) will tell you exactly why the model
failed to load (missing file, format mismatch, shape mismatch, etc).

> ⚠️ **Render free/standard instances spin down when idle.** The first
> request after inactivity ("cold start") may take 30–60 seconds while the
> instance boots and the model loads. This is normal — the frontend will
> just show the loading animation a bit longer on that first request.

## 4. Frontend Deployment (Netlify)

### Step 1 — Set your backend URL
Open `frontend/script.js` and update this line near the top with your actual
Render URL:
```javascript
const DEFAULT_API_URL = "https://deepshield-ai-v3.onrender.com";
```
(You can also skip this and just set it later from the app's **Settings**
page — it's saved in the browser via `localStorage` and takes priority over
the default.)

### Step 2 — Deploy to Netlify
**Option A — Drag & drop:**
1. Go to [app.netlify.com](https://app.netlify.com)
2. Drag the `frontend/` folder onto the Netlify dashboard
3. Done — Netlify hosts it instantly

**Option B — Git-based deploy:**
1. Go to Netlify → **Add new site** → **Import an existing project**
2. Connect your GitHub repo
3. Configure:
   - **Base directory:** `frontend`
   - **Build command:** *(leave empty — this is static HTML/CSS/JS)*
   - **Publish directory:** `frontend`
4. Deploy

Your frontend will be live at something like:
https://deepshield-ai.netlify.app

### Step 3 — Confirm the API URL in the app
Open the deployed Netlify site → **Settings** page → confirm/paste your
Render backend URL → click **Save** → click **Test Connection**. You should
see: `✅ Connected: DeepShield AI v3 running`.

## 5. API Reference

### `GET /`
Health check.
```json
{ "status": "DeepShield AI v3 running" }
```

### `GET /health`
Detailed health/diagnostics.
```json
{
  "status": "ok",
  "model_loaded": true,
  "model_error": null,
  "device": "cpu",
  "timestamp": "2026-07-16T12:00:00Z"
}
```

### `POST /scan`
Multipart form upload, field name `file`.
```bash
curl -X POST https://deepshield-ai-v3.onrender.com/scan \
  -F "file=@/path/to/image.jpg"
```
Response:
```json
{
  "result": "REAL",
  "confidence": 98.72,
  "explanation": "The model found strong evidence of natural image characteristics consistent with an authentic, unaltered photograph.",
  "filename": "image.jpg"
}
```

## 6. CORS

The backend enables CORS for all origins (`*`) on all routes via
`flask-cors`, so the Netlify-hosted frontend (or any origin) can call the
Render backend without CORS errors. No proxy or extra configuration needed.

## 7. Local Development (optional)

**Backend:**
```bash
cd backend
python -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate
pip install -r requirements.txt
python app.py
```
Runs on `http://0.0.0.0:5000` by default (or `$PORT` if set).

**Frontend:**
Just open `frontend/index.html` directly in a browser, or serve it with any
static file server. Set the API URL in the Settings page to point at your
local backend (e.g. `http://127.0.0.1:5000`) while testing locally, then
switch it back to your Render URL before sharing/presenting.

## 9. Things only you can confirm

Two details depend entirely on how you trained/saved your model, and I could
not verify them without your training script:

1. **Class index order** (`CLASS_NAMES` in `app.py`) — currently
   `["FAKE", "REAL"]`. Flip it to `["REAL", "FAKE"]` if your training
   pipeline indexed classes the other way.
2. **Checkpoint format** — `app.py` auto-detects full model vs. state_dict vs.
   wrapped checkpoint, but if loading fails, check the Render deploy logs for
   the exact `model_error` message and share it so the loading logic can be
   adjusted precisely to your file.

Everything else in this project is complete and deployable as-is.
